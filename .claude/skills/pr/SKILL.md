# pr

Open a pull request on GitHub from the current branch into `main`. Uses the `gh` CLI; falls back to printing the PR-create URL if `gh` isn't available.

## Pre-flight

Run in parallel:

1. `gh --version` — is `gh` installed? If not, print the manual URL at the end instead.
2. `git branch --show-current` — branch to open the PR from.
3. `git rev-parse --abbrev-ref --symbolic-full-name @{u} 2>/dev/null` — is the branch pushed?
4. `git log main..HEAD --oneline` — every commit that will land in the PR.
5. `git diff --stat main..HEAD` — file-level summary.
6. `gh pr list --head <branch> --json number,url 2>/dev/null` — does a PR already exist for this branch?

Then:

- If branch is **not pushed**, run the `push` skill first (or tell the user to). Do not attempt to open a PR for a branch that doesn't exist on the remote.
- If the current branch is **`main`** or **`master`**, refuse — PRs go *from* feature branches *to* main, never main into itself.
- If a PR **already exists** for this branch, print its URL and stop. Use `gh pr edit` separately if the user wants to edit it.
- If there are **zero commits ahead of main**, tell the user there's nothing to open and stop.

## Title

Pull the subject from the most recent commit on the branch. If the branch contains a single commit, that subject IS the title. If multiple commits exist, prefer the most descriptive `feat:`/`fix:` subject; fall back to deriving a title from the branch name (kebab-case → "Sentence case").

Keep titles ≤ 70 chars.

## Body

Compose using this template:

```markdown
## Summary

<2-4 bullet points covering the change, derived from the commit subjects and any OpenSpec proposal.md>

## OpenSpec change

<If openspec/changes/<branch-name>/ exists, link the proposal: `openspec/changes/<name>/proposal.md`. If the change has been archived (openspec/changes/archive/YYYY-MM-DD-<name>/), link that.>

## Commits

<auto-generated, one bullet per commit on the branch:
- `<short-sha>` <subject>>

## Test plan

- [ ] <auto-generate concrete checklist items from the change's tasks.md if present>
- [ ] Otherwise: list what you'd manually verify before merging
```

If an OpenSpec change exists for this branch, prefer its proposal.md "What Changes" section verbatim (lightly trimmed) for the Summary bullets — the proposal is the source of truth for the PR's intent.

## Create

```bash
gh pr create \
    --base main \
    --head "$(git branch --show-current)" \
    --title "<title>" \
    --body "$(cat <<'EOF'
<rendered body from template above>
EOF
    )"
```

**No AI signatures in the PR body.** Same rule as commits: no `🤖 Generated with...`, no `Co-Authored-By`, no trailing AI attribution.

## Draft vs ready

Default to a regular (non-draft) PR. If the user says "draft" or the branch's last commit subject contains `wip`, `WIP`, or `[draft]`, add `--draft`.

## After creation

Print the PR URL and a one-line summary of what's in it (commit count + file count from the pre-flight stats).

## Fallback (no gh)

If `gh` isn't installed, print the manual PR-create URL:

```
https://github.com/<owner>/<repo>/compare/main...<branch-name>?expand=1
```

Derive `<owner>/<repo>` from `git remote get-url origin`. Tell the user the title and body you would have used so they can paste them in.
